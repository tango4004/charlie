# Charlie Protocol Specification

**Version:** 0.3-draft  
**Status:** Draft  
**Project:** Charlie — Universal Agent Connector  

---

## Overview

Charlie is a protocol for asynchronous communication between LLM agents and workspaces via MCP (Model Context Protocol). It defines a minimal packet format, a two-pipe transport contract with port-based multiplexing, and a standard MCP interface.

Charlie is transport-agnostic: the buffer may use FIFO, hashmap, or any storage internally. The protocol does not prescribe implementation details.

---

## Packet Format

Every unit of data in Charlie is a **packet**. Packets are immutable in transit — no component between sender and receiver may alter the payload.

```json
{
  "job_id":    "string — persistent, links request to response",
  "version":   "string — protocol version, e.g. '0.1'",
  "routing":   "string — destination identifier (workspace_id or token)",
  "reply_to":  "string — source identifier (token or workspace_id)",
  "ts":        "ISO8601 — creation timestamp, used for TTL enforcement",
  "seq":       "integer — chunk sequence number, 0-based",
  "seq_total": "integer — total chunks for this job_id",
  "port":      "string — logical channel within a pipe (e.g. 'stdout', 'stderr', 'help')",
  "payload":   "string — opaque, immutable, application-defined"
}
```

### Field Rules

- **job_id**: generated by the bridge on submission. Immutable for the lifetime of the request/response pair. Unique across the system.
- **version**: must match supported versions of receiver. Mismatch → reject with error.
- **routing**: destination identifier. On inbound: workspace_id (routes packet to workspace). On outbound: token (routes result back to originating client). The merger stamps this field based on schema connections.
- **reply_to**: source identifier. On inbound: token of the originating client. On outbound: workspace_id of the responding workspace. The workspace swaps `routing` and `reply_to` when constructing the response packet. Enables stateless return routing without shared route tables.
- **ts**: UTC timestamp of packet creation. Buffer uses this to enforce TTL. Expired packets are dropped silently and an error is written to the outbound pipe on port `stderr`.
- **seq** / **seq_total**: chunking fields. If payload fits in one packet: seq=0, seq_total=1. Assembly at receiver: collect all seq 0..seq_total-1 for a given job_id, ordered by seq. Missing chunk → error.
- **port**: logical channel identifier carried inside the packet. Port semantics are opaque to the protocol layer — the buffer and router do not interpret port values. Only the sending and receiving endpoints assign meaning to port names. Well-known port names (`stdout`, `stderr`, `help`) are conventions, not protocol requirements.
- **payload**: opaque string. Neither bridge, buffer, nor transport layer interprets or modifies this field. Only the sending and receiving endpoints understand its content.

### Routing Example

Inbound (client → workspace):
```json
{ "routing": "bash", "reply_to": "t_doo", "port": "stdin", ... }
```

Workspace response (workspace → client):
```json
{ "routing": "t_doo", "reply_to": "bash", "port": "stdout", ... }
```

The merger on the inbound side stamps `reply_to` with the source pipe identity. The workspace swaps the fields and sets the appropriate `port`. The outbound router reads `routing` to deliver the result to the correct client. No shared state required.

## Two-Pipe Transport Contract

Charlie uses two physical pipes per connection. Logical channels (stdout, stderr, help, or any application-defined channel) are multiplexed onto these pipes via the `port` packet field.

```
inbound pipe:   bridge → buffer → workspace   (client submits work)
outbound pipe:  workspace → buffer → bridge   (workspace delivers all output)
```

### Rules

- There are exactly two pipes. All logical channels share the outbound pipe, distinguished by `port`.
- The buffer does not inspect or filter by `port`. It delivers all packets from the outbound pipe to the bridge as-is.
- The bridge demultiplexes by `port` after reading from the outbound pipe.
- Port semantics are defined by the endpoints, not by the protocol layer. The buffer is port-agnostic.
- Workspaces may write packets with any `port` value concurrently. Order within a port is preserved per seq; cross-port ordering is not guaranteed.
- Clients poll the outbound pipe and filter by `port` as needed.

### Buffer Endpoints (reference)

```
POST /inbound/{job_id}   — write packet to inbound pipe (any port)
POST /outbound/{job_id}  — write packet to outbound pipe (any port)
GET  /outbound/{job_id}  — read next packet(s) from outbound pipe
```

Logical channel separation is the responsibility of the bridge and workspace, not the buffer.

## MCP Interface (Charlie Connector)

The Charlie protocol has **two primitives**:

### `call`

Single entry point for all operations.

```
call(token, workspace_id, payload) → job_id
```

Returns job_id immediately. Payload is submitted to workspace's inbound pipe on port `stdin`.

### `get`

Poll the outbound pipe for a given job, filtered by port.

```
get(token, job_id, port) → payload (or empty if not ready)
```

`port` is a string naming the logical channel: `"stdout"`, `"stderr"`, or any connector-defined port name. Discovery and documentation are also ports: `get(token, job_id, "help")` returns the node's documentation.

> Two primitives. `call` submits. `get` reads any port by name — including help, stdout, stderr, or any future channel. The MCP tool list never changes regardless of port count or workspace type.

### Help as a protocol requirement

Every node with an address token **must** respond to `get(token, job_id, "help")`. Help is not special — it is a port like any other. A workspace that does not implement the `"help"` port is non-compliant.

### MCP connector wrapper (optional)

The MCP connector is the entry point — it has no address token of its own from the client's perspective. The client connects to it directly via MCP, not through Charlie routing. Therefore, the MCP connector exposes a hardcoded `get_help` tool: a static response with no job_id, no routing, no call required. This is not a protocol primitive — it is the bootstrap for clients that have no token yet.

## Token Model

A **token** is the client's credential. It is issued by the Allocator (separate component).

```json
{
  "token":      "t_xxxxxxxxxxxx",
  "workspaces": [
    {"id": "bbs",  "name": "Bravo BBS"},
    {"id": "bash", "name": "Bash executor"}
  ]
}
```

- One token may grant access to multiple workspaces.
- Each workspace_id maps to a specific executor behind the buffer.
- Token policies (TTL, revocation, rate limits) are defined by the Allocator, not by Charlie.
- The bridge validates the token on every call. Invalid or expired token → error on outbound pipe, port `stderr`.

---

## Workspace Contract

Any workspace that participates in Charlie must implement the following interface:

- Accept packets from buffer via **inbound pipe**
- Write results to buffer via **outbound pipe** using appropriate `port` values
- Serve documentation via port **`help`** (static .md file returned on request)
- Must not modify packet fields other than payload and port in response packets
- Must use the same job_id in response packets as received in the request

Workspace internals are not prescribed by Charlie.

---

## TTL and Cleanup

- Buffer checks `ts` of each packet against configured TTL (default: configurable per workspace).
- Expired packets are dropped. A synthetic error packet is written to the outbound pipe on port `stderr` with job_id preserved.
- Clients that never call `get` will eventually have their packets TTL-expired.
- Partial chunks (seq_total not reached before TTL) are dropped as a unit.

---

## Versioning

- Protocol version is carried in every packet (`version` field).
- Components must reject packets with unsupported versions.
- Minor version increments: backward compatible.
- Major version increments: breaking change, requires explicit upgrade.

---

## Token Symmetry

Every participant in Charlie — client or workspace — is identified by a token. The protocol does not distinguish between them. A token holder can be:

- An MCP connector (LLM agent: Claude, GPT, Grok...)
- A workspace (bash executor, SQL interface, BBS node...)
- Another MCP connector acting as a destination (agent-to-agent)

The switch routes by the `routing` field to any token. Whether the destination is a workspace or another agent is invisible to the routing layer.

Agent-to-agent: set `routing = "t_gpt"`. The packet arrives in GPT's inbound pipe directly.

BBS is a separate concern: a message store and broadcast layer for pub/sub and async delivery across nodes. It is not required for direct point-to-point messaging in Charlie.

---

## What Charlie Is Not

- Charlie is not a security layer. Token validation is minimal. Deep security is the Allocator's domain.
- Charlie is not a workflow engine. It does not chain jobs, branch, or retry.
- Charlie does not define workspace internals. BBS, bash, SQL — all are valid workspaces.
- Charlie does not define BBS or inter-node broadcast (that is Alpha/Bravo).
